# macOS Application Host

This document provides a technical overview of the macOS application host module. This module serves as the native entry point and container for the Flutter application on macOS, bridging the gap between the operating system's application lifecycle and the Flutter UI framework.

## Overview

The macOS host is a standard native macOS application that embeds a Flutter view. Its primary responsibilities are:

1.  **Application Initialization**: Responding to the operating system's launch signals and setting up the main application object.
2.  **Window Management**: Creating and managing the native `NSWindow` that will display the Flutter content.
3.  **Flutter Engine Bootstrapping**: Initializing the `FlutterViewController`, which loads and runs the Dart code.
4.  **Plugin Registration**: Connecting any required native plugins (Platform Channels) to the Flutter engine.

The entire native host is composed of two key Swift files: `AppDelegate.swift` and `MainFlutterWindow.swift`.

## Architecture

The architecture follows a standard containment pattern where native macOS components wrap and host the Flutter runtime.

```mermaid
graph TD
    subgraph macOS Native Layer
        A[NSApplication] --> B(AppDelegate);
        A --> C(MainFlutterWindow);
    end

    subgraph Flutter Bridge
        C -- hosts --> D(FlutterViewController);
        D -- registers --> E(Generated Plugins);
    end

    subgraph Flutter Layer
        D -- renders --> F[Flutter UI (Dart)];
    end

    B -- Manages App Lifecycle --> A;
```

1.  The `NSApplication` singleton, managed by AppKit, is the core of the application process.
2.  Our custom `AppDelegate` receives lifecycle callbacks from `NSApplication`.
3.  The `MainFlutterWindow` (an `NSWindow`) is loaded at startup, typically from `MainMenu.xib`.
4.  `MainFlutterWindow` creates and embeds a `FlutterViewController`, which is the object responsible for managing the Flutter engine instance.
5.  The `FlutterViewController` loads the compiled Dart code and renders the Flutter UI.

## Key Components

### `AppDelegate.swift`

This file defines the application's delegate, which is the entry point for the macOS application.

```swift
@main
class AppDelegate: FlutterAppDelegate {
  // ...
}
```

-   **`@main`**: This attribute marks `AppDelegate` as the application's entry point. The system will create an instance of this class when the application launches.
-   **`FlutterAppDelegate`**: The class inherits from `FlutterAppDelegate` instead of the standard `NSObject <NSApplicationDelegate>`. `FlutterAppDelegate` is a convenience class provided by the Flutter framework that handles much of the boilerplate setup for a Flutter application.

#### Lifecycle Methods

-   **`applicationShouldTerminateAfterLastWindowClosed(_:) -> Bool`**
    This delegate method determines the app's behavior when its last window is closed. By returning `true`, we configure the standard macOS behavior where closing the final window terminates the application process. To create an app that continues running in the background or in the dock (like a menu bar app), you would return `false`.

-   **`applicationSupportsSecureRestorableState(_:) -> Bool`**
    Returning `true` opts into macOS's state restoration feature. This allows the system to save and restore the window's size, position, and other UI state across application launches, providing a better user experience.

### `MainFlutterWindow.swift`

This file defines the custom `NSWindow` subclass that hosts the Flutter user interface.

```swift
class MainFlutterWindow: NSWindow {
  override func awakeFromNib() {
    // ...
  }
}
```

The primary logic resides in the `awakeFromNib()` method, which is called by the AppKit framework after the window has been loaded from its Interface Builder file (`MainMenu.xib`).

#### Initialization in `awakeFromNib()`

The setup process within `awakeFromNib` is critical and sequential:

1.  **Create a Flutter View Controller**:
    ```swift
    let flutterViewController = FlutterViewController()
    ```
    An instance of `FlutterViewController` is created. This controller manages the Flutter engine, renders the UI, and forwards input events.

2.  **Embed the View Controller**:
    ```swift
    self.contentViewController = flutterViewController
    ```
    The `flutterViewController` is set as the `contentViewController` of the `MainFlutterWindow`. This is the key step that places the Flutter view inside the native window, causing it to fill the window's content area.

3.  **Register Plugins**:
    ```swift
    RegisterGeneratedPlugins(registry: flutterViewController)
    ```
    This crucial function call wires up any plugins that require native macOS code. The `RegisterGeneratedPlugins` function is auto-generated by the Flutter build tools and contains the necessary code to initialize and register each plugin's platform channel handler with the Flutter engine. Without this, any calls from Dart to native plugins will fail.

## How to Extend

This module is the primary place to add custom native macOS functionality.

-   **Handling Custom App Events**: Add more `NSApplicationDelegate` methods to `AppDelegate.swift` to handle events like opening files, handling URLs, or managing application state.
-   **Platform Channel Implementation**: To add your own custom platform channels, you can implement the channel handlers directly within `AppDelegate` or `MainFlutterWindow`, or create new Swift files for better organization. Remember to register your new channel with the `flutterViewController.engine.binaryMessenger`.
-   **Modifying Window Behavior**: You can customize the window's appearance and behavior (e.g., style, title bar, transparency) by setting properties on `self` within `MainFlutterWindow.swift`.
