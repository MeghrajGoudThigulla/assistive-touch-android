1.5 Competitive Analysis & Failure Points

UI/UX Failures:

Icons are hard to see on screen, apps must be restarted every time you turn on your device, and have ridiculous ads for each keystroke to setup or alter the menu.
Several have icons hard to see on your screen — poor contrast, no dark mode done properly.
Bugs including the control dock disappearing when the keyboard is opened, missing overlay functions, and the assistive ball disappearing when going to the home screen.
The Settings option on Assistive Touch doesn't open the app configuration — instead it opens quick controls — confusing UX.
Every competitor UI looks like it was built in 2017. None use Material 3.

Performance Failures:

If the assistive touch is idle for 4 minutes it just disappears — ForegroundService instability.
Only supports Android versions 4.1 and up — outdated targeting.

Trust/Privacy Failures:

It's not clear exactly where the app stores its downloaded data.
This app requires you to approve a lot of permissions, which some users might be hesitant about.
No competitor clearly explains what each permission does before asking for it.

1. Executive Summary

Assistive Touch is a zero-login Android application that renders a persistent floating overlay button and a customisable 2-page action panel. The app gives users one-tap access to system navigation, device controls, and quick-action tools from anywhere on screen without interrupting their workflow.

The product is split into two runtime layers:

Flutter (Dart) — configuration UI, localisation, and persistent settings.
Kotlin native — foreground service, WindowManager overlay, gesture detection, and action execution.

Design Principle: Every action in the panel must be explicitly triggered by the user. The app never executes actions automatically, silently, or in the background.

2. Goals & Non-Goals
2.1  Goals
Provide a reliable, always-visible floating overlay button that works across all foreground apps.
Allow deep customisation of panel actions, floating button icon, size, opacity, and gesture bindings.
Deliver a clean, localised first-launch experience with simple, 2-3 screen onboarding and language selection.
Ensure transparent permission handling — every sensitive permission must be explained in-app before redirecting to system settings.
Target Play Store approval with zero policy violations on first submission.
App should feel instant (<100ms interaction latency) and lightweight (<50MB RAM usage).

2.2  Non-Goals (v1 Scope)
User accounts, login, or cloud sync.
Remote icon pack downloads or CDN assets.
Analytics or crash-reporting SDKs (added in a future phase with full Data Safety re-declaration).
Advertisements.
Auto-start on device boot.
Screen recording or screenshot (deferred to v1.1 after Play Store approval is confirmed).
Macro recording, automation sequences, or auto-click/auto-scroll functionality (permanently out of scope for this product line).

2.3 Design Principles, Competitive Mitigation & Targets

Your Winning Edge — Competitive Mitigation:

Floating button disappears when keyboard opens or after idle — your ForegroundService must never drop.
Icons are invisible against real wallpapers — your overlay must always be legible on any background using contrast-adaptive rendering.
Settings screen is confusing — your app has a clear separation between in-app config and overlay quick controls.
Ugly, outdated UI from 2015-2018 — you build with Material 3, dark-first, with micro-animations.
Excessive unexplained permission requests — you explain every permission before asking, in plain language.
Ads interrupting every setup step — your app is completely ad-free.
App needs manual restart after reboot — your ForegroundService handles this gracefully with a persistent notification.

UI/UX EXCELLENCE RULES:

1. Visual polish — Dark-first design, always. Every screen uses this exact palette: Background gradient: #0F172A → #020617, Floating button: #FFFFFF with subtle inner gradient, Accent: #3B82F6, Secondary text: #94A3B8, Shadow: rgba(0,0,0,0.25). Use flutter_animate for all transitions.
2. Overlay legibility — The floating button must always be visible regardless of wallpaper. Implement a dynamic shadow/glow ring that adapts — white button gets dark shadow on light backgrounds, and a subtle blue glow on dark backgrounds. This is the #1 visual differentiator.
3. Onboarding clarity — Permission explanation screens must be the most honest and clear of any app in this category. Each permission gets: what it does, what it does NOT do, a visual icon, and a single CTA. No dark patterns.
4. Zero friction customisation — Panel slot editing opens a bottom sheet with a searchable action list grouped by category (Navigation / System / Media / Display).
5. Micro-interactions — Every user action gets feedback: haptic on panel open, spring animation on button drag-snap, ripple on slot selection, smooth fade on settings toggles.

PERFORMANCE TARGETS — Hard Limits:

Interaction latency: under 100ms tap-to-action
Memory footprint: under 50MB RAM persistent
ForegroundService: must never drop — handle onTaskRemoved, onDestroy, and OEM battery kill signals
Use const constructors everywhere possible
No setState in large widget trees — use localised state only
Overlay button must remain visible when soft keyboard opens — handle WindowInsets in OverlayController.kt

3.2  OEM-Specific Notes
Samsung, Xiaomi, Oppo, Vivo — aggressive battery managers may kill background services. Overlay must run inside a ForegroundService with a persistent notification.
Android 12+ — overlay windows must set FLAG_NOT_FOCUSABLE where appropriate; WindowManager params must handle insets changes.
Android 13/14 — notification permission (POST_NOTIFICATIONS) must be requested at runtime for the foreground service notification.
Wi-Fi, Mobile Data, Airplane Mode, Location toggles — restricted by Android policy since API 29+. Fallback must open the relevant system Settings panel rather than toggling directly.

4. User Journeys
Journey A — First Launch
Trigger: App opened for the very first time (locale preference not saved).
1. App launches → Language Selection screen appears.
2. User selects language → locale saved, UI immediately re-renders in chosen language.
3. User lands on the Home screen.
4. Subsequent launches skip Language Selection and go directly to Home.

Journey B — Customise Panel
Trigger: User opens Customize Touch Panel from Home.
1. User views 2-page 3x3 grid (Main / Setting).
2. User taps a slot → Select Action bottom sheet appears.
3. User picks action → slot updates icon and label.
4. Configuration auto-persists; overlay panel reflects change immediately if running.

Journey C — Customise Floating Icon
Trigger: User opens Customize App Icon from Home.
1. Bundled icon grid displayed (4-column layout).
2. User selects icon → selection highlighted.
3. User taps Save (✓) → iconId persisted.
4. If overlay is running, icon updates live; otherwise on next overlay start.

Journey D — Settings
Trigger: User opens Settings from Home.
1. User changes any setting (gesture / size / opacity / theme / battery info).
2. Change applies instantly to the running overlay where applicable.
3. All settings persist across app restarts.

6. Architecture
6.1  High-Level Overview
Layer			Responsibility
Flutter (Dart)		Screens, localisation, SharedPreferences persistence, config editing, permission guidance
Kotlin Native		ForegroundService, WindowManager overlay lifecycle, gesture detection, action execution, EventChannel emissions
Bridge			MethodChannel (Flutter → Kotlin commands) + EventChannel (Kotlin → Flutter events)

6.2  Kotlin Native Components
- FloatingService: Hosts overlay lifecycle.
- OverlayController: Manages WindowManager views.
- FloatingButtonView: Custom View for drag/gestures.
- PanelOverlayView: 3x3 grid logic.
- ActionExecutor: Executes actions.
- AssistiveAccessibilityService: Executes global actions.
- DeviceAdminReceiver: Implements screen lock.
- BatteryMonitor: Tracks battery changes.

6.3  MethodChannel Contract (Flutter → Kotlin)
Methods: overlay.start(), overlay.stop(), overlay.updateConfig({config}), overlay.executeAction({actionId}), permissions.getState(), permissions.openOverlaySettings(), permissions.openAccessibilitySettings(), deviceAdmin.requestActivation().

6.4  EventChannel Contract (Kotlin → Flutter)
Events: permissionStateChanged, overlayStateChanged, batteryInfo, error.

7. Permissions & Fallbacks
Features requiring permissions: Floating overlay (SYSTEM_ALERT_WINDOW), Service (FOREGROUND_SERVICE), Navigation (Accessibility), Lock (Device Admin), Flashlight (Camera), System Toggles (restricted, fallback to Settings), Battery (broadcast), Notification (POST_NOTIFICATIONS).
All must fail gracefully when denied.

8. Data Model
8.1  Config Schema (shared_preferences): locale, panel.main, panel.setting, floatingButton.iconId, floatingButton.size, floatingButton.opacity, ui.backgroundColor, ui.darkMode, gesture.singleTap, gesture.doubleTap, gesture.longPress, animation.speed, features.showBatteryInfo.
8.2  Storage Strategy: v1 is local only (shared_preferences).

9. Error Handling & Edge Cases
Scenarios handled: Revoked permissions, Accessiblity disabled, Device Admin revoked, Service killed by OS, Rotation repositioning, Swiped from Recents, Action assigned to ungranted permission slot.
